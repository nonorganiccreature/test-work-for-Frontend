# 1. Анализ приложения по:

- качество кода: Наличие any в типизации ref'ов вместо описания типа задач. Отсутствие очищения событий из бразуерного API keypress, resize на unmounted, в случае если в приложении есть роутинг.

- архитектура: Отсутствие декомпозиции логики, декомпозиции большого компонента на более меньшие. Типы вынести в отдельную директорию. Лишние файлы от инициализации проекта.

- производительность: Возможно стоило бы заменить на более точный метод расчетов таймера, при помощи timestamp'ов requestAnimationFrame

- безопасность: ссылка на внешние ресурсы для картинки в AppHeader при помощи тэга object, стоит использовать если проверенный источник

# 2. Кратакое описание задач:

Вынес логику в composables. Декомпозиция по компонентам. Так как компоненты для частного случая, сделал через внутреннее состояние + defineModel пропсы для изменения данных логикой внутри компонентов, вместо внешней логики + emit'ов. Отдельная страница с задачам так, как если бы был роутинг. Локальное сосотояние в компоненте страницы, так как не использовалась pinia или любой другой менеджер состояний, где можно было бы изолировать всю логику и состояние для страницы задач. Стандартный линтер для исключения несовпадений в кодовой базе.
